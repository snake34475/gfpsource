# 功夫派 (GFP) 游戏源码架构分析文档

## 一、项目概述

**游戏名称**: 功夫派 (GongFuPai / Kung Fu School)
**开发商**: 淘米 (Taomee)
**游戏类型**: 2D 横版动作 MMORPG
**运行平台**: Adobe AIR 桌面端
**编程语言**: ActionScript 3 (AS3)
**源码规模**: ~1,581 个 `.as` 文件

---

## 二、源码范围说明

### 2.1 本仓库包含什么

这是**纯客户端**源码，包含两个 SWF 模块：

| 模块 | 路径 | 说明 |
|------|------|------|
| **clientcoredll** | `clientcoredll/` | 核心库 SWF（加密/解密、Crossbridge C 模块桥接） |
| **clientappdll** | `clientappdll/` | 游戏应用主模块（~1,528 个 AS 文件，所有游戏内容） |
| **xml 配置** | `xml/` | 服务器配置、Bean 注册表 |

### 2.2 本仓库**不包含**什么

| 缺失部分 | 说明 |
|----------|------|
| **后端服务器源码** | 战斗伤害计算公式、属性系统、掉落逻辑等核心运算都在服务端 |
| **com.gfp.core.\*** | 核心游戏引擎框架库（MainManager、MapManager、SocketConnection、CommandID 等） |
| **com.taomee.\*** | 淘米基础框架库（网络通信、数据结构、显示工具） |
| **服务器端配置** | 怪物属性表、技能数值表、装备数值表等 |

## 三、客户端-服务器架构

### 3.1 瘦客户端架构

本游戏采用**瘦客户端**架构，核心原则：

```
┌─────────────────────────────────────────────────────────┐
│                      服 务 器 端                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │ 伤害计算  │  │ 属性系统  │  │ 掉落逻辑  │  │ 验证逻辑 │ │
│  │ (攻-防)  │  │ (攻击/防御)│  │ (物品/经验)│  │ (反作弊) │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
│                        ↕ Socket TCP (端口 1863)           │
├─────────────────────────────────────────────────────────┤
│                      客 户 端 (本仓库)                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │ 输入处理  │  │ 动画播放  │  │ UI 显示   │  │ 音效播放 │ │
│  │ (按键/鼠标)│  │ (受击/移动)│  │ (血条/数字)│  │ (打击音效)│ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
└─────────────────────────────────────────────────────────┘
```

**关键结论**: 伤害计算公式（攻击力 - 防御力、暴击倍率等）**不在客户端**，而是由服务器计算后将结果下发给客户端显示。

### 3.2 服务器地址配置

从 `xml/Server.xml` 可以看到：

| 服务器类型 | 地址 | 端口 |
|-----------|------|------|
| 主游戏服 | `114.80.173.15` | 1863 |
| 电信登录服 | `kflogin.61.com` | - |
| 网通登录服 | `cnckflogin.61.com` | - |
| 资源服 | `gf.61.com` | - |

游戏共有 **54 个服务器**（如 "艾尔大陆"、"武圣大殿"、"一气学院" 等），按等级分区。

### 3.3 网络通信协议

- **传输层**: Socket TCP 二进制协议
- **消息路由**: 通过 `CommandID` 常量标识消息类型
- **数据格式**: `ByteArray` 二进制序列化
- **Bean 注册**: `xml/dll.xml` 注册了 **60+ 个命令监听器**

### 4.2 伤害计算流程（客户端视角）

客户端**不计算伤害**，只**接收并显示**服务器发来的伤害结果。完整流程：

```
1. 玩家按下攻击键
   └── 客户端发送技能使用请求到服务器

2. 服务器计算伤害
   ├── 读取攻击方属性（攻击力、暴击率、技能系数）
   ├── 读取防御方属性（防御力、闪避率、抗性）
   ├── 应用 Buff/Debuff 修正
   ├── 计算是否暴击
   ├── 计算最终伤害值
   └── 下发 BruiseInfo 到客户端

3. 客户端接收 BruiseInfo (BruiseCmdListener.as:67-156)
   ├── decHP: 伤害值（已计算好的最终结果）
   ├── hp: 受击后剩余血量
   ├── crit: 是否暴击 (布尔值)
   ├── critMultiple: 暴击倍率
   ├── decType: 伤害类型 (0=普通, 1=吸收, 等)
   ├── type: 受击动作类型 (0-8)
   ├── hitCount: 连击数
   ├── breakFlag: 是否打断
   └── skillID/skillLv: 技能信息

4. 客户端执行表现
   ├── 播放受击动画 (NetBruiseAction / NetKnockFlyAction)
   ├── 显示伤害数字 (NumberManager.showSub / showCrit)
   ├── 播放打击音效
   ├── 屏幕震动（暴击时）
   └── 更新血条
```

### 4.3 BruiseInfo 关键字段（伤害相关）

从 `BruiseCmdListener.as` 代码分析，服务器下发的 `BruiseInfo` 包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| `decHP` | int | **伤害值**（服务器已计算好） |
| `hp` | int | 受击后剩余 HP |
| `atkerID` | uint | 攻击者 ID |
| `userID` | uint | 受击者 ID |
| `crit` | Boolean | **是否暴击** |
| `critMultiple` | int | **暴击倍率** |
| `decType` | int | 伤害类型：0=扣血, 1=吸收 |
| `type` | int | 受击表现类型：0-8 不同动作 |
| `hitCount` | int | 连击数 |
| `breakFlag` | Boolean | 是否打断施法 |
| `skillID` | uint | 技能 ID |
| `skillLv` | uint | 技能等级 |
| `weaponEnchantedType` | int | 武器附魔类型 |
| `runDuration` | Number | 受击动作持续时间 |
| `stiffDuration` | Number | 僵直持续时间 |
| `pos` | Point | 受击位移目标位置 |

### 4.4 暴击表现

暴击的**计算在服务端**，客户端只负责表现：

```actionscript
// BruiseCmdListener.as:319-322
if(param2.crit)
{
   NumberManager.showCrit(param1, param2.decHP, param2.critMultiple);
}
```

暴击时额外触发：
- 屏幕震动 (`mapShock`, line 370-380)
- 特殊暴击数字显示

### 4.5 受击动作类型

| type 值 | 动作 | 说明 |
|---------|------|------|
| 0 | NetBruiseAction (40001) | 普通受击 |

## 五、地图移动系统分析

### 5.1 移动架构

```
玩家输入 (键盘/鼠标)
    └── 客户端发送移动请求
        └── 服务器广播位置到其他客户端
            └── MoveCmdListener / MouseMoveCmdListener 接收
                └── 执行移动动画
```

### 5.2 移动命令监听器

| 文件 | 监听命令 | 说明 |
|------|----------|------|
| `MoveCmdListener.as` | MOVE, PVP_MOVE, OGRE_MOVE, SUMMON_MOVE | 所有移动类型 |
| `MouseMoveCmdListener.as` | MOUSE_MOVE | 鼠标点击移动 |
| `JumpCmdListener.as` | - | 跳跃 |
| `StandCmdListener.as` | - | 站立/待机 |
| `PositionChangeCmdListener.as` | - | 位置变更 |
| `RideCmdListener.as` | - | 骑乘移动 |
| `LeaveMapCmdListener.as` | - | 离开地图 |

### 5.3 移动子类型 (SubType)

| 子类型 | 动作 ID | 说明 |
|--------|---------|------|
| WALK | 10002 | 行走 |
| RUN | 10003 | 跑动 |
| TELEPORT | 10011 | 瞬移/传送 |
| MOUSE_RUN | 9002 | 鼠标跑动 |
| MOUSE_WALK | 9001 | 鼠标行走 |

### 5.4 移动数据协议

服务器下发的移动数据包包含：

```
MOVE 命令:
  - userID: 玩家 ID
  - mapID: 地图 ID
  - pos (x, y): 目标坐标
  - direction: 朝向
  - subType: 移动类型 (走/跑)

PVP_MOVE 命令 (额外包含):
  - startTime: 移动开始时间戳
  - sequence: 序列号（用于 UDP 同步校验）
```

### 5.5 地图处理系统

| 目录 | 文件数 | 说明 |
|------|--------|------|
| `mapProcess/` | **521 个** | 每个地图专属逻辑 |

每个地图有独立的 `MapProcess_XXXXXX.as` 类，处理：

## 七、游戏启动流程

从 `MainEntry.as` (1,131 行) 分析：

```
1. 加载配置
   └── 读取 Server.xml、dll.xml

2. 连接服务器
   └── Socket 连接到游戏服务器 (IP:114.80.173.15:1863)

3. 登录验证
   └── 账号验证 → 选择服务器 → 选择角色

4. 初始化游戏系统
   ├── 加载 UI 资源
   ├── 注册 Bean/命令监听器
   ├── 初始化各管理器
   └── 加载角色数据

5. 进入地图
   └── 根据角色位置加载对应 MapProcess

6. 种族新手引导
   └── 根据角色类型（猴、兔、熊猫、龙、虎、猫、马、狼）播放对应引导
```

## 九、总结

1. **这是客户端源码**，不是完整的游戏源码
2. **伤害计算公式在服务端**，客户端只负责接收结果并播放动画/显示数字
3. **移动逻辑**：客户端发送移动意图，服务器广播位置，各客户端同步表现
4. **要看到完整的属性设计和伤害计算**，需要找到对应的**服务端源码**
5. 客户端代码仍然有很高的分析价值，可以理解：
   - 游戏整体架构和模块划分
   - 网络协议结构（通过 ByteArray 解析可以反推数据包格式）
   - UI/动画系统设计
   - 各功能模块的交互方式
